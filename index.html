Product Brief

Understand RetailFlow: what it does, who it serves, and what the migration is trying to achieve.

Executive Summary
RetailFlow is a mid-market B2C e-commerce platform serving over 280,000 active customers across Nigeria, Ghana, and Kenya. Founded in 2017, it was built as a PHP monolith on a single MySQL database. Revenue has grown 340% in four years, but the architecture has not scaled with it. The platform now suffers from cascading failures, deployment freezes, and an average incident resolution time of 6.4 hours. The board has approved a full migration to a microservices architecture hosted on AWS, with a hard go-live deadline that cannot move.

The Migration Goal
Decompose the monolith into eight independent microservices, each owning its own database. Services will communicate via REST APIs and asynchronous events through AWS SQS. The migration must preserve all existing customer data, maintain 99.5% uptime during cutover, and deliver a 60% improvement in checkout performance. The team has 90 days. This capstone places your group at Day 82 — eight days before go-live.

Known Pain Points Going Into Migration

  • 01
    Shared Database Coupling All eight modules currently share a single MySQL schema. The migration to separate databases has not been fully tested under concurrency. Inventory and Payments both write to the same `orders` table.
  • 02
    No Circuit Breakers If the Payment service degrades, the Checkout service has no fallback — it hangs indefinitely. This caused the 14-hour outage in March that cost ₦180M in lost GMV.
  • 03
    Inventory Race Conditions Under peak load (e.g., flash sales), multiple checkout sessions can reserve the same item simultaneously. The legacy system resolves this with a cron job — not real-time locking.
  • 04
    Async Event Bus Untested The new SQS-based event bus has never been load tested. Nobody knows what happens to Fulfillment and Notifications queues during a flash sale of 10,000 concurrent orders.
  • 05
    Data Migration Gaps Legacy customer records include ~4,200 orphaned orders (no matching user_id) and ~900 products with NULL category fields. These have not been resolved in the migration scripts.

Critical Failure Chains

These are the cascades your group must trace and document in Section 1 of the PQEB.

⚡ Chain 01 — Payment Cascade
Payment Service fails ↓
→ Cart cannot confirm checkout
→ Order Service never receives order.created event
→ Inventory reservation lock never releases
→ Fulfillment never dispatches
→ Customer receives no notification
⚡ Chain 02 — Inventory Race
Flash sale triggers 800 concurrent checkouts ↓
→ Inventory DB cannot lock fast enough
→ Overselling occurs (stock goes negative)
→ Order confirmations sent for unavailable items
→ Fulfillment fails silently
→ Manual refund queue overwhelmed
⚡ Chain 03 — SQS Queue Backup
Event bus saturates under load ↓
→ Fulfillment events delayed 15–40 mins
→ Notification Service sends stale status updates
→ Customer support call volume spikes
→ SLA breaches trigger penalty clauses with 3PL partners
⚡ Chain 04 — Auth Token Gap
Logout does not blacklist JWT token ↓
→ Stolen token remains valid until expiry (24 hrs)
→ Fraudulent orders placed on compromised accounts
→ Payment Service processes charges
→ No real-time fraud detection in new stack
Your Group's Mission: Map these chains in your Dependency Graph. For each chain, identify the exact API call, DB query, or event that is the single point of failure. This becomes your evidence in the Go/No-Go Dossier.

Seeded Logic Gaps — For Facilitator Reference

These are the deliberate gaps hidden in the roadmap above. Groups must independently identify and articulate them.

# Gap Title Where It Hides Why It's Dangerous
G1
Inventory tested at 10 users, not 10,000
The roadmap says "considered sufficient for launch" after 10 concurrent users — a flash sale drives 800+.
 Phase 2 · Inventory Service Overselling, negative stock, manual refund chaos
G2
No rollback plan for production data migration
Scheduled night-before with no rollback documentation. If the script fails at 3AM, go-live proceeds anyway.
 Phase 3 · Data Migration Irrecoverable data corruption with no fallback
G3
Tax calculation only confirmed for Lagos
RetailFlow serves Nigeria, Ghana, and Kenya. Tax logic for Abuja, Port Harcourt, Accra, and Nairobi is unverified.
 Phase 2 · Cart & Checkout Incorrect billing in 60%+ of geographic coverage
G4
Load testing self-assessed by Dev Team with no QA oversight
PM confirmed load testing is internal only. No independent QA engineer, no defined pass/fail threshold.
 Phase 3 · Load Testing Happy-path bias baked into the test design
G5
No parallel run — legacy decommissioned same night as go-live
Zero overlap between old and new system. No canary deployment. No traffic shadowing. Big bang cutover.
 Phase 3 · Go-Live Single failure point with 280,000 live users and ₦2.4B monthly GMV at risk
G6
JWT token blacklist not implemented
Auth Service does not blacklist tokens on logout. Compromised sessions remain valid for 24 hours.
 Phase 1 · Auth Service Active fraud window on every customer account post-logout
G7
Fulfillment only tested for Lagos and Abuja routes
3PL integration confirmed for two routes. RetailFlow ships to 28 states and 3 countries. Other routes untested.
 Phase 3 · Fulfillment Service Silent fulfillment failures in 90%+ of shipping destinations
